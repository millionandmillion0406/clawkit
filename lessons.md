# 工程纪律 · 远程部署铁规

> 每次改服务器代码前，照做。少一步都别动。

---

## 为什么需要

远程改代码最常见的翻车路径：sed 盲改 → 服务器炸 → 不知道哪行改错了。

不是工具问题，是流程问题。

## 五条铁规

### 1. 改前先读
动手改服务器代码前，先 `cat` 全文读完。不凭记忆操作。

### 2. 改后验语法
每次修改后立即跑对应语言的语法校验：
- **Python**: `python -m py_compile`
- **JS/TS**: `npx tsc --noEmit` 或 `npm run lint`
- **Go**: `go vet`
- **Rust**: `cargo check`
- **Shell**: `shellcheck`

### 3. 部署验存活
根据项目类型选择验证方式：
- **Web 服务** → `curl` 健康检查，HTTP 200
- **CLI 工具** → `--help` + 最小化功能测试
- **脚本** → 本地 dry-run
- **前端** → 打开页面确认正常

### 4. 一次一处
一次只改一个地方。验证通过后再改下一个。不可一口气改多处。

### 5. 留记录
每次改动写进工作台文件或提交信息。改了什么、怎么改的、结果。

### 6. 留退路
改之前备份或 `git stash`。改坏了要有办法一秒还原。

## Antipatterns（反例）

| ❌ 不要 | ✅ 应该 |
|--------|--------|
| sed 盲改远程文件 | 本地编辑 → SFTP 上传 → curl 验证 |
| Token 写死在代码里 | `os.environ.get()` |
| 部署完直接走人 | `curl` + 功能测试 |
| 凭记忆改配置 | 每次改动写记录 |
| 一口气改三处一起部署 | 改一处、验一处、再改下一处 |
| 单人 review 自己 | 用审查模板请至少一个其他 Agent 审一遍 |
